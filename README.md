## 1. 任务内容
&emsp;&emsp;CCL（China National Conference on Computational Linguistics）为中国计算语言学大会，会议组织单位为中国中文信息学会。CCL是全国最权威、最具影响力、规模最大的NLP会议之一，它聚焦于中国境内各类语言的智能计算和信息处理，为研讨和传播计算语言学最新学术和技术成果提供了广泛的高层次交流平台。  

&emsp;&emsp;近年来，我国在信息无障碍领域已从“倡导推动”进入“制度保障”阶段。《中华人民共和国无障碍环境建设法》明确要求提升信息交流与数字服务的无障碍水平，“十四五”残疾人保障和发展规划以及数字中国、公共服务数字化相关政策亦明确提出要加强无障碍信息产品与服务供给。  

&emsp;&emsp;在这一背景下，字幕不再只是“体验优化功能”，而是面向听障群体与老年群体获取音视频信息的**基础性公共服务能力**。特别是在医院就诊、银行柜台办理、行政服务大厅咨询、工作会议等“高即时性”与“高信息密度”的真实办事场景中，听障用户面临着独特的挑战：  

1.	**社交时差（Latency）带来的交流断裂：** 字幕生成的延迟直接导致交流节奏的脱节，“延迟多一秒，用户就要多尴尬一秒”。

2.	**关键信息丢失的风险：** 在嘈杂环境下，相比于语法的通顺，听障用户更在意金额、药名、证件号、时间等**核心关键词**是否准确。

&emsp;&emsp;然而，当前学术界与产业界仍缺乏一个面向无障碍应用需求、统筹考量“可读性”、“核心信息准确度”与“响应速度”的统一评测基准。  

&emsp;&emsp;在我国现有约 2,700 万听力残疾人以及大量老年听力减退人群的背景下，字幕是其获取音视频信息的核心无障碍手段。随着短视频、直播、在线会议及远程协作的快速发展，“AI 字幕”已从“辅助功能”逐渐演变为信息无障碍基础设施。与传统的语音识别任务不同，真实应用场景中的字幕系统不仅要求较高的转写准确率，还对**时间轴对齐、合理断句与行宽控制、标点与语气表达、多人说话场景下的说话人区分、背景音乐与噪声条件下的鲁棒性、幻觉文本抑制**提出了更高要求。  

&emsp;&emsp;本评测聚焦于从“语音/视频输入”到“面向听障用户与字幕依赖用户阅读的结构化字幕文档生成”的完整链路能力，系统评测模型在真实无障碍应用场景中的综合表现。为了更贴近真实应用场景，本次评测将任务划分为“**PC端**”与“**手机端**”两个平行赛道。每个赛道均包含两个层次的子任务（Tracks），其中 Track 2 的目标不是生成“技术上正确”的文本，而是生成在信息获取层面真正**可用（准确捕捉关键信息）**、**可理解（结构化）**、**即时（低延迟）** 的无障碍字幕内容。  

### 1.1	赛道设置  

#### 赛道A：PC端  

&emsp;&emsp;场景定义：模拟云端服务、桌面端软件或高性能工作站场景。  

&emsp;&emsp;资源限制：不限制模型大小与计算资源。  

&emsp;&emsp;目标：探索当前技术在信息无障碍字幕生成任务上的性能上限，追求极致的转写准确率与结构化生成的可用性。  

#### 赛道B：手机端  

&emsp;&emsp;场景定义：模拟听障用户适用手机、AR眼镜等移动设备进行实时交流的场景。  

&emsp;&emsp;资源限制：对模型体积、内存占用与实时性提出明确约束要求。  

&emsp;&emsp;目标：评估在资源受限、功耗敏感、算力有限的条件下，系统是否仍能提供低延迟、高可用的无障碍字幕服务。  

### 1.2	子任务设置  

两个赛道均包含以下两个子任务：  

#### Track 1：基础字幕生成（Foundation Track）  

&emsp;&emsp;输入：音频或视频  

&emsp;&emsp;输出：带时间戳的逐字或逐句字幕  

&emsp;&emsp;评测重点：转写准确性、时间轴对齐精度、噪声与复杂场景鲁棒性  

&emsp;&emsp;说明：该 Track 面向传统 ASR / 多模态语音识别系统，旨在为评测提供稳健的基础对比平台。在手机端赛道中，额外考核在低资源下的识别稳定性。  

#### Track 2：结构化可读字幕生成（Structured Track）  

&emsp;&emsp;输入：音频或视频  

&emsp;&emsp;输出：面向人类阅读的结构化字幕文档，包括合理断句、行宽与字幕块切分、标点符号、时间轴、说话人区分、尽量避免幻觉文本。  

&emsp;&emsp;评测重点： 在保证可读性的同时，重点考核在嘈杂、工作会议等场景下对核心关键词（实体词、数字、指令）的捕捉能力，以及字幕生成的时间紧凑性（低延迟）。  

&emsp;&emsp;说明：该 Track 旨在评测系统从“识别语音”到“生成可发布字幕内容”的综合生成能力。在手机端赛道中，重点考核如何利用小模型实现复杂的结构化生成与幻觉抑制。   

## 2. 评测数据  

### 2.1 数据总体规模与原则  

&emsp;&emsp;本评测计划构建一个总规模约 **30–50 小时** 的多场景真实语音/视频测试集。    

&emsp;&emsp;其中一部分核心子集（约 10–15 小时）将提供更高粒度的结构化标注与核心关键词标注、说话人信息，用于Track 2 评测和人工主观评测，其余数据提供标准转写、时间轴。

&emsp;&emsp;两个赛道将使用相同的数据集内容，以保证性能指标的可比性。  

### 2.2 数据集构成  

|数据子集|场景描述|难点特征|占比|
|---|---|---|---|
|**Set A：新闻与演讲**|新闻联播、TED演讲、正式会议|发音标准、无背景音、单人独白|20%|
|**Set B：影视与综艺**|电视剧片段、综艺节目、电影|背景音乐干扰 (BGM)、音效干扰、情绪化表达|30%|
|**Set C：生活与真实交流场景（探店、办事、访谈、Vlog、随手拍等）**|户外直播、街头采访、探店视频，将尽可能引入模拟办事与真实对话型场景数据。|环境噪声（风声/车流）、远场收音、设备收音差|30%|
|**Set D：多人会议**|真实会议录音、圆桌讨论|多人重叠语音、抢话、口语填充词（Uh/Ah）|20%|

### 2.3 标注层级  

#### 基础标注：

- **文本标注**： 包含口语词保留版本。

- **时间戳标注**： 精确到毫秒级的句子/词级开始与结束时间。  

#### 高质量子集额外提供：

-	**说话人标识**： 针对多人场景，标注Speaker ID。  

-	**可读性优化后的参考字幕版本**  

-	**核心关键词标注**  

### 2.4 样例数据

#### 输入音频：原始的无字幕场景音频文件。

#### 输出标准：
* **基础字幕**：
仅包含切分好的文本内容以及对应的毫秒级时间戳（`start_time_ms` 和 `end_time_ms`）。
```json
{
    "text": "请您拿着这张表，去对面的三号窗口办理。",
    "start_time_ms": 29760,
    "end_time_ms": 35705
},
```

* **结构化字幕**：
  在基础字幕的文本中，准确插入了结构化标签：
  * `<speakerN>`：区分不同的说话人（如表演者、顾客等）。
  * `<key>...</key>`：标记语句中的核心关键信息。
```json
{
    "text": "<speaker1>请您拿着这张表，去对面的<key>三号窗口</key>办理。",
    "start_time_ms": 29760,
    "end_time_ms": 35705
},
```

## 3. 评价标准（更新中）

&emsp;&emsp;为了全面衡量真实应用场景中的字幕系统的质量，我们将采用“客观指标 + 关键信息指标 + 主观体验 + 资源效率指标”相结合的评测方式。  

### 3.1 核心评价指标（适用于两个赛道）  

#### Track 1（自动评测）  

1. **文本准确性**：基于字错误率（CER），剔除标点符号干扰进行纯文本计算；  

2. **时间轴偏差**：基于生成时间戳与人工时间戳的起始点和结束点的平均绝对误差（MAE）； 

#### Track 2（自动 + 人工混合评测）  

1. **自动评测指标**

    1. **文本准确性**：基于字错误率（CER），剔除标点符号干扰进行纯文本计算；  

    2. **时间轴偏差**：基于生成时间戳与人工时间戳的起始点和结束点的平均绝对误差（MAE）； 
   
    3. **说话人准确率**：基于字符级对齐，评估多人场景下说话人角色（Speaker ID）切分与匹配的正确性。  

3. **人工评测指标（抽样）**

   1. **核心信息识别准确率**

   2. **可读性（1–5 分）**：由于字幕是给“人”看的，断句和展示效果至关重要。我们将邀请专业字幕组人员或语言学家进行人工打分（1-5分制）。  

   3. **断句合理性**：字幕切分是否符合语法逻辑？是否在不该断的地方断开了（如把一个词切断）？单行字数是否过长导致阅读困难？

   4. **标点与语气匹配度**：语气词（疑问、感叹）的标点是否准确反映了说话人的情感？

### 3.2 赛道特异性指标

#### 赛道 A：PC端  

- **首字延迟（Latency）**：作为重要参考指标，但不设立硬性门槛。

#### 赛道 B：手机端

- **实时率 (RTF, Real Time Factor)**：音频处理耗时需小于音频本身的时长（即 RTF < 1.0）。

- **模型体积 (Model Size)**：提交的所有模型权重文件（包含 ASR 与后处理模型）总大小 **≤ 300 MB**。

- **内存占用 (Peak Memory)**：运行时峰值内存占用 **≤ 800 MB**。

最终成绩采用公开权重的加权综合评分。

## 6. 提交要求与技术报告

### 撰写技术报告

- 报告可由中文或英文撰写。  

- 报告统一使用CCL 2024的论文模板。论文模板下载链接：http://cips-cl.org/static/CCL2024/downloads/ccl2023_template.zip 。  

- 报告正文不得超过6页，参考文献页数不限。  

- 报告应至少包含以下四个部分：模型介绍、评测结果、结果分析与讨论和参考文献。

- 会议投稿需统一使用LaTeX模板。

- 由于本次会议采用双盲审稿，作者姓名和单位不能出现在投稿的论文中。因此，作者的自引不可采用“我们提出”的方式，而是用“作者名字提出…”。

- 不符合要求的论文将不经过完整的审稿流程而直接被拒稿。

- 统一 SRT / JSON 等指定格式
  
- 参赛队伍需在提交时明确选择参加 “PC端赛道” 或 “手机端赛道”（可同时参加，但需分别提交模型和结果）。

## 7. Baseline 系统

本评测将提供可复现的 baseline 系统。  （代码在github目录上）

### 通用 Baseline

-	Track 1：基于 FunASR 开源中文语音识别工具链构建。

-	Track 2：结合规则后处理与大模型的关键词增强机制。

### 轻量化 Baseline

-	提供一套基于轻量化网络架构的端侧方案。

-	Track 2 部分将演示如何使用小参数量语言模型或 N-gram 语言模型进行低延迟的标点预测与结构化断句，以满足移动端资源限制。
